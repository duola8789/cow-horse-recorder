## Context

当前小程序使用微信云开发架构：
- 云数据库存储用户数据（users、clock_records、holidays）
- 云函数处理业务逻辑（login、clock、getStats、updateRecord 等）
- 云开发环境即将到期（2026-04-21），云数据库收费 19.9/月

约束条件：
- 单用户应用（仅开发者本人使用）
- 数据量小（预计 50 年数据 < 2MB）
- 小程序 Storage 限制 10MB
- 需要零成本运行

技术栈：
- Taro 3.x + React + TypeScript
- dayjs 用于时间处理
- 微信小程序 Storage API

## Goals / Non-Goals

**Goals:**
- 完全本地化数据存储，不依赖任何云服务
- 删除所有云函数，业务逻辑前端化
- 保持所有现有功能（打卡、统计、设置）
- 节假日数据智能同步（12月自动拉取次年）
- 零运行成本

**Non-Goals:**
- 不做数据迁移（用户重新开始使用）
- 不支持多设备同步
- 不支持云端备份
- 不做离线缓存优化（Storage 本身就是离线的）

## Decisions

### 决策 1: 数据存储方案 - 本地 Storage

**选择：** 使用微信小程序 `wx.storage` 作为唯一数据存储

**理由：**
- 容量足够（10MB >> 实际 2MB）
- 持久化，不会因清理缓存而丢失
- 完全免费
- 离线可用

**替代方案：**
- ❌ 云数据库：需要付费
- ❌ COS 对象存储：增加复杂度，读写慢
- ❌ 自建服务器：需要运维成本

### 决策 2: 数据结构 - 扁平化 Map

**选择：** 使用日期作为 key 的 Map 结构

```typescript
{
  "2026-04-14": { startTime: "09:30", endTime: "18:30", ... },
  "2026-04-13": { startTime: "09:25", endTime: "18:35", ... }
}
```

**理由：**
- O(1) 查询性能
- 易于增删改查
- JSON 序列化友好
- 不需要按年分片（50年才 2MB）

**替代方案：**
- ❌ 数组结构：查询需要遍历，性能差
- ❌ 按年分片：增加复杂度，无必要

### 决策 3: 时间存储格式 - HH:mm 字符串

**选择：** 存储 "HH:mm" 格式字符串（如 "09:30"）

**理由：**
- 避免时区转换问题
- 存储空间小
- 易于展示和编辑
- 避免 Date 对象序列化问题

**替代方案：**
- ❌ Date 对象：序列化后变字符串，时区混乱
- ❌ 时间戳：需要频繁转换，不直观

### 决策 4: 节假日同步策略 - 智能按需

**选择：** 仅在 12 月且没有次年数据时请求 API

**理由：**
- 最小化网络请求
- 节假日数据一年更新一次
- API 失败时降级到默认规则（周一到周五）

**流程：**
```
App 启动
  ↓
检查当前月份
  ↓
是 12 月？
  ├─ 是 → 检查有次年数据？
  │         ├─ 无 → 请求 API
  │         └─ 有 → 跳过
  └─ 否 → 跳过
```

**替代方案：**
- ❌ 每次启动都请求：浪费流量
- ❌ 内置数据每年更新：需要发版
- ❌ 手动触发同步：用户体验差

### 决策 5: 架构分层 - 三层结构

**选择：** UI → Service → Storage 三层架构

```
Pages (UI 层)
  ↓ 调用
Services (业务逻辑层)
  ↓ 调用
Storage Wrapper (数据层)
  ↓ 调用
wx.storage (微信 API)
```

**理由：**
- 职责分离，易于测试
- 业务逻辑复用（多个页面共享）
- Storage 层统一错误处理
- 未来易于扩展（如增加云备份）

**替代方案：**
- ❌ 页面直接操作 Storage：逻辑重复，难维护
- ❌ 单一 Service 层：职责不清晰

### 决策 6: 去除登录认证

**选择：** 完全移除用户认证逻辑

**理由：**
- 单用户应用，无需区分用户
- 数据存本地，天然隔离
- 简化代码和流程

**影响：**
- 删除 `login` 云函数
- 删除 `_openid` 字段
- 删除 `users` 表逻辑

## Risks / Trade-offs

### 风险 1: iOS 时区兼容性警告

**风险：** `dayjs.tz()` 在 iOS 下有格式警告

**缓解：** 使用 `dayjs.utcOffset(8)` 代替 `dayjs.tz('Asia/Shanghai')`

**验证：** POC 测试已确认结果正确，仅有警告不影响功能

---

### 风险 2: Storage 写入失败

**风险：** 手机存储空间不足导致写入失败

**缓解：**
- 所有写操作包裹 try-catch
- 失败时提示用户清理空间
- 不影响读操作

**概率：** 极低（数据量仅 2MB）

---

### 风险 3: 节假日 API 不可用

**风险：** 第三方 API 挂掉或限流

**缓解：**
- 失败时降级到默认规则（周一到周五为工作日）
- 不影响核心打卡和统计功能
- 仅统计准确性略降（会把法定假日当工作日）

**影响：** 低，用户可接受

---

### 风险 4: 小程序被删除导致数据丢失

**风险：** 用户删除小程序，数据永久丢失

**缓解：**
- 文档说明数据存储位置
- 未来可增加导出功能（可选）

**接受：** 用户已明确表示可接受

---

### Trade-off 1: 性能 vs 复杂度

**选择：** 全量加载 clock_records 到内存

**权衡：**
- ✅ 查询快速（O(1)）
- ✅ 代码简单
- ❌ 启动时加载全部数据（但仅 2MB，可忽略）

**决策：** 接受，性能足够好

---

### Trade-off 2: 离线优先 vs 云端同步

**选择：** 完全离线，不做云端同步

**权衡：**
- ✅ 零成本
- ✅ 隐私保护
- ✅ 架构简单
- ❌ 换设备数据丢失
- ❌ 不能多设备使用

**决策：** 接受，符合单用户场景

---

### Trade-off 3: 节假日准确性 vs API 依赖

**选择：** 依赖第三方 API，失败时降级

**权衡：**
- ✅ 节假日数据准确（国务院官方）
- ✅ 自动更新
- ❌ 需要网络请求
- ❌ API 可能失效

**决策：** 接受，有降级方案

## Migration Plan

### 阶段 1: 开发（2 小时）

1. 创建基础设施（30 分钟）
   - `src/utils/storage.ts` - Storage 封装
   - `src/constants/storage.ts` - Key 常量
   - `src/types/data.ts` - 数据类型

2. 创建业务服务（60 分钟）
   - `src/services/clockService.ts`
   - `src/services/statsService.ts`
   - `src/services/holidayService.ts`

3. 修改 UI 层（45 分钟）
   - 打卡页面：调用 ClockService
   - 统计页面：调用 StatsService
   - 设置页面：直接操作 Storage

4. 清理和测试（15 分钟）
   - 删除云函数目录
   - 删除云开发配置
   - 完整功能测试

### 阶段 2: 配置（5 分钟）

1. 微信公众平台配置域名白名单
   - 添加 `https://timor.tech`

2. 删除 POC 测试页面（可选）

### 阶段 3: 发布

1. 提交代码
2. 小程序后台提交审核
3. 审核通过后发布

### 回滚方案

如果发现严重问题：
1. 恢复 git 到迁移前的 commit：`git checkout v1.0.0-before-migration`
2. 重新部署云函数
3. 云数据库数据未删除，可继续使用

**建议：迁移前打 tag**
```bash
git tag v1.0.0-before-migration
git push origin v1.0.0-before-migration
```
